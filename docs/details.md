# 詳細設計書

このドキュメントは、ToDo Webアプリ「きょう、なにする？」の詳細設計書である。
基本設計書をもとに、開発者向けに実装・テスト・運用・運用に関する詳細内容を記載する。

---

## 1. はじめに

この詳細設計書の目的・対象読者・適用範囲を明記する章です。基本設計書との役割の違い（基本設計書が「何を作るか」を決めるのに対し、詳細設計書は「どう作るか」を決める）を記載します。このドキュメントを読めば、担当開発者が迷わずコーディングに着手できる状態を目指します。また、参照する関連ドキュメント（要件定義書・基本設計書・APIドキュメントなど）へのリンクも記載します。

---

## 2. 画面詳細設計

基本設計書で定義した各画面について、表示内容・操作・バリデーション・エラー表示を詳細に定義する章です。開発者がこの章を見るだけで、ある画面を最初から最後まで実装できるように記載します。

### 2-1. 画面詳細の記載方針

各画面ごとに以下の項目を記載します。画面が多い場合は、画面ごとにファイルを分けてもかまいません。

- **画面ID・画面名・URLパス**：どの画面かを特定する情報
- **表示条件**：この画面を表示するための前提（例：ログイン済みであること）
- **表示項目一覧**：画面に表示されるすべてのテキスト・入力欄・ボタン・画像など
- **バリデーション（入力チェック）**：各入力欄に対する検証ルールとエラーメッセージ
- **処理の流れ**：ボタンを押したときに何が起こるか（API呼び出し→成功時/失敗時の動作）
- **画面遷移先**：操作後に移動する画面

### 2-2. 画面詳細（例：ログイン画面）

**画面ID**：SC-001　**画面名**：ログイン画面　**URLパス**：/login

**表示項目**

| 項目名 | 種類 | 表示ラベル | 初期値 | 備考 |
|--------|------|-----------|--------|------|
| メールアドレス | テキスト入力欄 | メールアドレス | 空 | type="email" |
| パスワード | パスワード入力欄 | パスワード | 空 | type="password"、文字は非表示 |
| ログインボタン | ボタン | ログイン | — | フォーム送信 |
| パスワードを忘れた方へ | リンク | パスワードを忘れた方はこちら | — | /password_reset に遷移 |

**バリデーション**

| 入力項目 | 検証ルール | エラーメッセージ |
|---------|-----------|---------------|
| メールアドレス | 未入力不可、メールアドレス形式であること | 「メールアドレスを入力してください」「正しいメールアドレス形式で入力してください」 |
| パスワード | 未入力不可、8文字以上 | 「パスワードを入力してください」「パスワードは8文字以上で入力してください」 |

**処理の流れ**

1. ユーザーがメールアドレスとパスワードを入力し「ログイン」ボタンを押す
2. フロントエンドでバリデーションを実行する（失敗した場合はエラーメッセージを表示して処理を止める）
3. バリデーション通過後、`POST /api/sessions` にリクエストを送信する
4. レスポンスが成功（200）の場合：セッションを保存し、ダッシュボード（/dashboard）に遷移する
5. レスポンスが認証エラー（401）の場合：「メールアドレスまたはパスワードが正しくありません」と画面上に表示する

## 3. API詳細設計

基本設計書のAPI一覧をもとに、各APIのリクエスト・レスポンスの詳細仕様を定義する章です。フロントエンドとバックエンドの開発者が、この章を見ながら実装を進められるレベルまで具体化します。

### 3-1. API詳細の記載方針

各APIごとに以下の項目を記載します。

- **API ID・エンドポイント・HTTPメソッド**
- **概要**：何をするAPIか
- **認証**：アクセスにログインが必要かどうか
- **リクエストパラメータ**：送信するデータの項目名・型・必須かどうか
- **レスポンス**：成功時・失敗時それぞれのHTTPステータスとデータ構造
- **エラーケース**：起こりうるエラーとそのレスポンス

### 3-2. API詳細（例：ユーザー登録）

**API ID**：API-001　**エンドポイント**：`POST /api/users`　**認証**：不要

**概要**：新しいユーザーをシステムに登録する。登録成功後、確認メールを送信する。

**リクエストボディ（JSON）**

| パラメータ名 | データ型 | 必須 | 説明 |
|------------|---------|------|------|
| email | string | 必須 | ユーザーのメールアドレス。一意であること |
| password | string | 必須 | パスワード。8文字以上64文字以内 |
| name | string | 必須 | ユーザーの表示名。100文字以内 |

**リクエスト例**

```json
{
  "email": "user@example.com",
  "password": "securePassword123",
  "name": "山田 太郎"
}
```

**レスポンス（成功時：201 Created）**

```json
{
  "id": 1,
  "email": "user@example.com",
  "name": "山田 太郎",
  "created_at": "2025-01-01T00:00:00Z"
}
```

**エラーレスポンス一覧**

| HTTPステータス | エラーコード | メッセージ例 | 発生条件 |
|--------------|------------|------------|---------|
| 400 | VALIDATION_ERROR | 「入力内容を確認してください」 | パラメータ不正・必須項目未入力 |
| 409 | EMAIL_ALREADY_EXISTS | 「このメールアドレスはすでに登録されています」 | 同一メールアドレスが既存 |
| 500 | INTERNAL_SERVER_ERROR | 「システムエラーが発生しました」 | サーバー内部エラー |

## 4. データベース詳細設計

基本設計書のテーブル定義をさらに詳細化し、インデックス・制約・初期データなども含めて定義する章です。開発者がこの章を見てマイグレーションファイル（データベースの構造変更を記録するファイル）を作成できるレベルの情報を記載します。

### 4-1. テーブル詳細定義

各テーブルについて、カラム名・データ型・制約（NOT NULL、UNIQUE、外部キーなど）・インデックス・初期データを記載します。

**usersテーブル詳細**

| カラム名 | データ型 | NULL | UNIQUE | デフォルト | 説明 |
|---------|---------|------|--------|-----------|------|
| id | BIGINT | 不可 | ○（主キー） | 自動採番 | ユーザーを一意に識別するID |
| email | VARCHAR(255) | 不可 | ○ | なし | ログイン用メールアドレス |
| password_digest | VARCHAR(255) | 不可 | — | なし | bcryptでハッシュ化したパスワード |
| name | VARCHAR(100) | 不可 | — | なし | 表示名 |
| role | VARCHAR(20) | 不可 | — | 'user' | 権限（'user' または 'admin'） |
| created_at | TIMESTAMP | 不可 | — | CURRENT_TIMESTAMP | レコード作成日時 |
| updated_at | TIMESTAMP | 不可 | — | CURRENT_TIMESTAMP | レコード更新日時 |

**インデックス**

| インデックス名 | 対象カラム | 種類 | 目的 |
|--------------|---------|------|------|
| users_pkey | id | PRIMARY KEY | 主キー。レコードの一意識別 |
| index_users_on_email | email | UNIQUE INDEX | メールアドレスの重複防止・ログイン検索の高速化 |

### 4-2. マイグレーション方針

データベースの構造変更をどのように管理するかの方針を記載します。マイグレーションツール（例：Flyway、Active Record Migrationsなど）の選定、命名規則、ロールバック手順などを定めます。変更履歴をコードとして管理することで、チーム全員が同じDB構造を再現できるようにします。

## 6. ログ設計

システムの動作状況を記録するログの種類・出力先・フォーマットを定義する章です。障害発生時の原因調査や、不正アクセスの検知に役立てるため、「何のログを・どこに・どんな形式で・いつまで保持するか」を決めておくことが重要です。

### 6-1. ログ種別と方針

| ログ種別 | 出力対象・タイミング | 出力先 | 保存期間 |
|---------|-----------------|--------|---------|
| アクセスログ | HTTPリクエストのたびに記録（URL・ステータスコード・処理時間） | Webサーバーログ | 90日 |
| アプリケーションログ | 処理の開始・終了、重要な分岐、外部API呼び出し結果 | アプリログファイル | 30日 |
| エラーログ | 例外発生時（スタックトレースを含む） | エラーログファイル | 180日 |
| 監査ログ | データの作成・更新・削除、ログイン・ログアウト | 専用テーブルまたはファイル | 1年 |

### 6-2. ログフォーマット

ログの1行あたりの出力フォーマットを統一して定義します。ログを機械的に解析する場合はJSON形式が扱いやすく、人が直接読む場合はテキスト形式が読みやすいです。以下の例を参考に、プロジェクトに合ったフォーマットを決めます。

```
[ログレベル] [日時] [処理ID] [メッセージ]
例）[INFO] 2025-01-01T12:00:00Z [req-abc123] POST /api/sessions 200 120ms
```

## 7. テスト設計

開発した機能が正しく動くことを確認するためのテストの種類・範囲・方法を定義する章です。すべての機能に対してどんなテストを行い、何をもって「合格」とするかの基準（テスト完了条件）を記載します。テスト設計を事前に行うことで、抜け漏れのない品質確認ができます。

### 7-1. テスト種別と方針

| テスト種別 | 目的 | 担当 | 実施タイミング |
|-----------|------|------|-------------|
| 単体テスト | 関数・メソッド単位での動作確認 | 開発者 | 実装中・都度 |
| 結合テスト | 複数のモジュールが連携して正しく動くかの確認 | 開発者 | 機能実装完了後 |
| システムテスト | システム全体の動作確認（シナリオベース） | テスト担当者 | リリース前 |
| 受け入れテスト | クライアント・ユーザーによる最終確認 | クライアント | リリース前 |
| 回帰テスト | 修正・追加によって既存機能が壊れていないかの確認 | 開発者 | バグ修正・機能追加後 |

### 7-2. テストケース（例：ログイン機能）

各機能に対するテストケースを、正常系（うまくいくケース）と異常系（失敗するケース）に分けて記載します。

| テストID | テスト内容 | 入力値 | 期待する結果 | 正常/異常 |
|---------|-----------|--------|------------|---------|
| TC-001 | 正しい情報でログインできる | 正しいメール・パスワード | ダッシュボードに遷移する | 正常系 |
| TC-002 | パスワードを間違えるとエラーになる | 正しいメール・誤ったパスワード | エラーメッセージが表示される | 異常系 |
| TC-003 | メールアドレスが未入力のときバリデーションエラーになる | 空・任意のパスワード | 入力欄の下にエラーが表示される | 異常系 |
| TC-004 | 存在しないメールアドレスではログインできない | 未登録メール・任意のパスワード | エラーメッセージが表示される | 異常系 |

## 8. 環境変数・設定値一覧

アプリケーションが動作するために必要な設定値（接続情報・APIキー・フラグなど）を一覧で管理する章です。設定値をコードに直接書かず環境変数で管理することで、セキュリティ向上とデプロイの柔軟性が高まります。実際の値（パスワード・秘密鍵など）はここには書かず、別途シークレット管理ツールで管理します。

| 変数名 | 説明 | 設定例（ダミー値） | 必須 |
|--------|------|----------------|------|
| DATABASE_URL | データベースの接続先URL | postgres://user:pass@localhost:5432/myapp | 必須 |
| SECRET_KEY_BASE | セッションや暗号化に使う秘密鍵 | （ランダムな64文字以上の文字列） | 必須 |
| SMTP_HOST | メール送信サーバーのホスト名 | smtp.sendgrid.net | 必須 |
| SMTP_PORT | メール送信サーバーのポート番号 | 587 | 必須 |
| SMTP_USER | メール送信用のユーザー名 | apikey | 必須 |
| SMTP_PASSWORD | メール送信用のパスワード | （SendGridのAPIキー） | 必須 |
| APP_BASE_URL | アプリのベースURL（メール内リンクなどに使用） | https://example.com | 必須 |
| LOG_LEVEL | ログの出力レベル（debug / info / warn / error） | info | 任意 |

## 9. デプロイ・リリース手順

本番環境へのリリース作業の手順を記載する章です。コードのデプロイ手順、データベースのマイグレーション手順、リリース前後の確認作業（スモークテスト）、問題が起きたときのロールバック手順を定めます。手順を文書化しておくことで、誰がリリース作業をしても同じ結果になるようにします。

| ステップ | 作業内容 | 担当 | 備考 |
|---------|---------|------|------|
| 1 | 本番ブランチへのマージを確認する | 開発者 | mainブランチが最新であること |
| 2 | ステージング環境で最終確認を行う | テスト担当者 | 合格後のみ本番リリース |
| 3 | メンテナンスページを表示する（必要な場合） | インフラ担当 | DBマイグレーション実行中 |
| 4 | データベースのマイグレーションを実行する | インフラ担当 | バックアップ取得後に実施 |
| 5 | アプリケーションをデプロイする | インフラ担当 | デプロイコマンドを実行 |
| 6 | 主要機能の動作確認（スモークテスト）を行う | 開発者 | ログイン・主要画面の表示確認 |
| 7 | メンテナンスページを解除する | インフラ担当 | 確認完了後 |
| 8 | 関係者にリリース完了を通知する | PM | Slackやメールで連絡 |

## 10. 用語集

この詳細設計書で使われる専門用語・略語・開発固有の言葉を説明する章です。詳細設計書は実装に近い内容を扱うため、フレームワーク固有の用語や、チーム内でのみ使われる呼称が登場することがあります。新しいメンバーが合流した際にも参照できるよう、用語を充実させておきます。

| 用語 | 読み方 | 意味・説明 |
|------|--------|-----------|
| インデックス | いんでっくす | データベースで特定のカラムを高速に検索するために作る索引のこと |
| マイグレーション | まいぐれーしょん | データベースの構造変更をコードとして記録・実行する仕組み |
| バリデーション | ばりでーしょん | 入力値が正しい形式・範囲かどうかを検証すること |
| シリアライズ | しりありーず | オブジェクトや構造体をJSON等の文字列形式に変換すること |
| スタックトレース | すたっくとれーす | エラー発生時に、どのコードを経由して問題が起きたかを示す情報 |
| スモークテスト | すもーくてすと | リリース後に主要機能だけ素早く確認する最低限の動作確認テスト |
| 冪等性（べきとうせい） | べきとうせい | 同じ操作を何度繰り返しても、結果が変わらない性質のこと |
| ロールバック | ろーるばっく | デプロイやDB変更が失敗したとき、元の状態に戻すこと |
| 外部キー | がいぶきー | 別のテーブルのレコードを参照するためのカラム。テーブル間を繋ぐ |
| （追加する用語） | （読み方） | （説明） |
