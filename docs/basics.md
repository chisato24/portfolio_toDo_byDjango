# 基本設計書

このドキュメントは、ToDo Webアプリ「きょう、なにする？」の基本設計書である。
要件定義書をもとに、システムの全体的な設計方針と構造を記載する。

---

## 1. はじめに

本書では、アプリ開発における以下の内容を記載する。

- ドキュメントの対象者
- ドキュメントの適用範囲（どのシステム・機能を対象とするか）
- 関連ドキュメント（要件定義書・詳細設計書など）への参照

---

## 2. システム全体構成

システムを構成する要素（フロントエンド・バックエンド・データベース・外部サービスなど）の関係を整理する章です。各要素がどのように連携しているかを図（システム構成図）と合わせて説明します。図がない場合でも、文章と表で「何がどこに存在し、どう繋がるか」を明確に示すことが重要です。

### 2-1. システム構成図

システム全体のアーキテクチャを視覚的に示す図を貼り付ける箇所です。ブラウザ・Webサーバー・アプリケーションサーバー・データベース・外部APIなどの要素と、その接続関係を矢印で表した図を用意します。図は draw.io・Mermaid・Figmaなどのツールで作成すると管理しやすくなります。

### 2-2. 構成要素一覧

| 構成要素 | 役割・概要 | 使用技術・サービス |
|---------|-----------|-----------------|
| フロントエンド | ユーザーが操作するブラウザ画面の表示・操作処理 | （例：React） |
| バックエンド | ビジネスロジックの処理・APIの提供 | （例：Node.js） |
| データベース | データの永続的な保存・管理 | （例：PostgreSQL） |
| Webサーバー | HTTPリクエストの受付・静的ファイルの配信 | （例：Nginx） |
| 外部API | 決済・認証・メールなどの外部サービスとの連携 | （例：Stripe） |
| インフラ | サーバーやネットワークの基盤 | （例：AWS EC2） |

## 3. 機能設計

要件定義書で定義した機能要件を、どのように実現するかを設計する章です。各機能の処理の流れ（入力→処理→出力）、関係する画面やデータ、使用するAPIなどを記載します。この章を読むことで、開発者が「この機能をどう作るか」のイメージを持てる状態にすることが目標です。

### 3-1. 機能設計一覧

| 機能ID | 機能名 | 処理の概要 | 入力 | 出力 | 関連画面 |
|--------|--------|-----------|------|------|---------|
| F-001 | ユーザー登録 | 入力情報をバリデーション後、DBに保存し確認メールを送信 | メールアドレス、パスワード | 登録完了メッセージ | 登録画面 |
| F-002 | ログイン | 入力情報とDBの値を照合し、一致すればセッションを発行 | メールアドレス、パスワード | セッショントークン | ログイン画面 |
| F-003 | データ一覧取得 | ログインユーザーに紐づくデータをDBから取得して返す | ユーザーID（セッション） | データ一覧（JSON） | データ一覧画面 |
| （追加） | | | | | |

### 3-2. 処理フロー

各機能の処理の流れをフローチャートまたは番号付きの手順で記載する箇所です。「ユーザーが操作する→システムが処理する→結果を返す」という一連の流れを、分岐条件（例：入力が正しい場合／正しくない場合）も含めて書きます。シーケンス図（Mermaidなど）を使うと視覚的に整理しやすくなります。

---

## 4. 画面設計

システムに存在するすべての画面の構成・遷移・レイアウト方針を定義する章です。どの画面に何が表示され、どのボタンを押すと次の画面に移動するかを整理します。ワイヤーフレームや画面遷移図を添付するとより明確になります。

### 4-1. 画面遷移図

ユーザーが画面をどのように移動するかを図で示す箇所です。ログイン前後で見られる画面が変わる場合は、それぞれのルートを分けて表現します。矢印で「どの操作をするとどの画面に移動するか」を示します。

### 4-2. 画面一覧と設計概要

| 画面ID | 画面名 | URLパス | 主な表示内容 | 主な操作・ボタン | 備考 |
|--------|--------|---------|------------|---------------|------|
| SC-001 | ログイン画面 | /login | メールアドレス・パスワード入力欄 | ログインボタン、パスワードを忘れた方へ | 未ログイン時のみアクセス可 |
| SC-002 | ダッシュボード | /dashboard | サマリー情報・最新データ | 各機能へのナビゲーション | ログイン必須 |
| SC-003 | データ一覧 | /items | データの一覧テーブル・検索フォーム | 検索、新規登録、編集、削除 | ページネーションあり |
| SC-004 | データ登録・編集 | /items/new, /items/:id/edit | 入力フォーム | 保存ボタン、キャンセルボタン | バリデーションあり |
| SC-005 | 管理者画面 | /admin | ユーザー一覧・設定項目 | ユーザー編集・削除 | 管理者権限のみ |

## 5. データベース設計

システムで扱うデータをデータベースのどのテーブルにどう保存するかを設計する章です。テーブルの一覧、各テーブルの項目（カラム）とデータ型、テーブル同士の関係（リレーション）を記載します。この設計が後の詳細設計書やプログラム実装の土台になります。

### 5-1. テーブル一覧

| テーブル名（英語） | テーブルの日本語名 | 概要 |
|-----------------|----------------|------|
| users | ユーザー | システムを利用するユーザーの情報を管理する |
| sessions | セッション | ログイン状態を管理するセッション情報を保持する |
| （例：posts） | （例：投稿） | （例：ユーザーが作成したデータを管理する） |

### 5-2. テーブル定義

各テーブルの項目名、データ型、必須かどうか、初期値、説明を一覧で記載します。後の詳細設計書でさらに細かく定義しますが、ここでは主要なカラムを網羅しておきます。

**usersテーブル（ユーザー）**

| カラム名 | データ型 | NULL | 初期値 | 説明 |
|---------|---------|------|--------|------|
| id | INTEGER | 不可 | 自動採番 | ユーザーを一意に識別するID |
| email | VARCHAR(255) | 不可 | なし | ログインに使用するメールアドレス |
| password_digest | VARCHAR(255) | 不可 | なし | ハッシュ化されたパスワード |
| name | VARCHAR(100) | 不可 | なし | ユーザーの表示名 |
| role | VARCHAR(20) | 不可 | 'user' | ユーザーの権限（user / admin） |
| created_at | DATETIME | 不可 | 現在日時 | レコード作成日時 |
| updated_at | DATETIME | 不可 | 現在日時 | レコード更新日時 |

### 5-3. テーブル関連図（ER図）

テーブル同士の関係（どのテーブルがどのテーブルを参照するか）を図で示す箇所です。「1対多（1人のユーザーが複数の投稿を持つ）」「多対多（ユーザーと商品がお気に入りで繋がる）」などの関係を矢印で表現します。Mermaid記法や draw.io で作成した図を貼り付けます。

## 6. API設計

フロントエンドとバックエンドがデータをやりとりするためのAPI（接続口）の設計を記載する章です。どのURLに、どんな方法（GET・POST・PUT・DELETEなど）でアクセスすると、どんなデータが返ってくるかを定義します。詳細なリクエスト・レスポンスの仕様は詳細設計書に委ねますが、ここでは全体のAPI一覧を整理します。

### 6-1. API一覧

| API ID | HTTPメソッド | エンドポイント（URL） | 機能の概要 | 認証必要 |
|--------|------------|-------------------|-----------|---------|
| API-001 | POST | /api/users | ユーザーの新規登録 | 不要 |
| API-002 | POST | /api/sessions | ログイン（セッション発行） | 不要 |
| API-003 | DELETE | /api/sessions | ログアウト | 必要 |
| API-004 | GET | /api/items | データ一覧の取得 | 必要 |
| API-005 | POST | /api/items | 新規データの登録 | 必要 |
| API-006 | PUT | /api/items/:id | 既存データの更新 | 必要 |
| API-007 | DELETE | /api/items/:id | 既存データの削除 | 必要 |

## 7. セキュリティ設計

要件定義書で定めたセキュリティ要件を、どのように実装するかを設計する章です。認証・認可の仕組み、通信の暗号化方法、入力値の検証（バリデーション）、攻撃への対策（SQLインジェクション・XSSなど）を記載します。「なぜその対策が必要か」という理由も合わせて書くと、開発者全員が意識を共有できます。

| セキュリティ項目 | 設計方針・実装方法 |
|---------------|----------------|
| 認証方式 | セッションベース認証。ログイン成功時にサーバー側でセッションIDを発行する |
| パスワード管理 | bcryptを使用してハッシュ化して保存。平文のまま保存しない |
| 通信暗号化 | 全ページでHTTPS（TLS 1.2以上）を使用する |
| CSRF対策 | フォーム送信時にCSRFトークンを検証する |
| XSS対策 | ユーザーからの入力値を画面に表示する際はエスケープ処理を行う |
| SQLインジェクション対策 | プレースホルダ（パラメータ化クエリ）を使用し、SQL文を直接組み立てない |
| アクセス制御 | ロール（role）に基づいてアクセスできるページ・APIを制限する |

## 8. インフラ・環境設計

システムを動かすためのサーバー構成や環境（開発・ステージング・本番）の設計を記載する章です。どのクラウドサービスを使うか、サーバーの台数や役割分担、ドメインやSSL証明書の取得方法、環境ごとの設定の違いなどを整理します。

### 8-1. 環境一覧

| 環境名 | 目的 | URL（例） | 備考 |
|--------|------|----------|------|
| 開発環境（local） | 開発者個人のPC上で動作確認 | http://localhost:3000 | 本番データを使わない |
| ステージング環境 | 本番公開前の最終確認・受け入れテスト | https://staging.example.com | 本番に近い構成 |
| 本番環境 | 実際のユーザーが使用する環境 | https://example.com | 高可用性・バックアップ必須 |

### 8-2. インフラ構成

使用するクラウドサービスやサーバーの種類、ネットワーク構成（VPC、サブネットなど）、ロードバランサーの有無、CDNの利用などを記載します。図（インフラ構成図）を添えると、全体のネットワーク構成が把握しやすくなります。

## 9. エラー・例外処理設計

システムで想定されるエラーの種類と、それぞれに対してどのような対応をするかを定義する章です。ユーザーに表示するエラーメッセージの文言、システムログへの記録方法、エラー発生時の画面遷移などを統一的に設計します。エラー処理を事前に設計しておくことで、開発時の実装ばらつきを防げます。

| エラー種別 | HTTPステータスコード | ユーザーへの表示内容 | システム側の対応 |
|-----------|-------------------|------------------|---------------|
| 入力値エラー | 400 Bad Request | 「入力内容に誤りがあります。確認してください。」 | バリデーションエラー詳細をレスポンスに含める |
| 認証エラー | 401 Unauthorized | 「ログインが必要です。」 | ログイン画面にリダイレクト |
| 権限エラー | 403 Forbidden | 「このページへのアクセス権限がありません。」 | ログを記録する |
| 未検出エラー | 404 Not Found | 「お探しのページが見つかりません。」 | 404専用ページを表示 |
| サーバーエラー | 500 Internal Server Error | 「システムエラーが発生しました。しばらくお待ちください。」 | エラー内容をログに記録し、担当者に通知 |

## 10. 用語集

この基本設計書で使われる専門用語・略語を説明する章です。設計書は要件定義書よりも技術的な内容が増えるため、開発経験が浅いメンバーや非エンジニアが読む際に参照できるよう、用語と意味の対応表を用意します。

| 用語 | 読み方 | 意味・説明 |
|------|--------|-----------|
| API | エーピーアイ | アプリケーション同士がデータをやりとりするための接続口・仕様 |
| ER図 | イーアールず | テーブル（データの表）同士の関係を図で表したもの |
| エンドポイント | えんどぽいんと | APIにアクセスするためのURL |
| カラム | からむ | データベースのテーブルにおける列（項目）のこと |
| セッション | せっしょん | ログイン状態など、一時的なユーザー情報をサーバーで保持する仕組み |
| バリデーション | ばりでーしょん | 入力されたデータが正しい形式・範囲かどうかを検証すること |
| リレーション | りれーしょん | データベースのテーブル同士が外部キーで繋がっている関係 |
| ロールバック | ろーるばっく | データベース操作が途中で失敗したとき、変更前の状態に戻す処理 |
| （追加する用語） | （読み方） | （説明） |
